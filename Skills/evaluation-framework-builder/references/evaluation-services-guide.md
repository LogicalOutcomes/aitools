# Evaluation Process Guide

This guide covers **how** to run evaluation projects at LogicalOutcomes. For the **why** behind our approach, see [Evaluation Principles](evaluation-principles.md). For the canonical plan structure, see [plan-structure.md](plan-structure.md).

---

## Before You Begin

**Start with the Evaluation Planner:** [evalplanner.logicaloutcomes.net](https://evalplanner.logicaloutcomes.net/)

Enter the program details, and the tool generates a first-draft evaluation plan. This gives you:

- A logic model tailored to the program
- Evaluation questions matched to the organization's needs
- A data collection plan prioritizing participant tracker data
- A reporting schedule appropriate for the organization's capacity

Use this as your working document, adapting it as you learn more about the context.

---

## Project Roles

Every evaluation needs these four functions filled. One person can fill multiple roles, but responsibilities must be explicit.

| Role | Typical Person | Key Responsibilities |
|------|----------------|---------------------|
| **Sponsor** | Senior manager at the organization | Champions evaluation at leadership level; ensures resources; removes barriers |
| **Liaison** | Program manager | Primary contact with you; manages internal scheduling; translates between evaluation and program perspectives |
| **Project Owner** | You (lead evaluator) | Accountable for deliverables; defines methodology; makes strategic decisions |
| **Project Manager** | You or coordinator | Monitors timeline; coordinates data collection; keeps the project moving |

**For smaller organizations:**

- Sponsor and Liaison roles are often combined
- The minimum: someone must OWN the evaluation (strategic) and someone must MANAGE it (operational)

## The Three Drivers

Before designing anything, understand *why* this evaluation is happening:

| Driver | What It Means | Your Approach |
|--------|---------------|---------------|
| **Funder requirement** | Funder requires evaluation as a funding condition | Tie methods directly to funder questions; minimize burden; demonstrate funding is well-spent |
| **Management concern** | Board or leadership wants to investigate specific issues | Focus on the specific concern; may need targeted interviews; connect findings to management decisions |
| **Staff improvement** | Program staff want to improve their services | Focus on participant feedback and process analysis; team problem-solving; practical, actionable findings |

Most evaluations address multiple drivers. The Evaluation Planner helps structure plans that serve all three.

## The Logic Model

Every evaluation uses a standard logic model with two 3-column tables:

**Table 1: Program resources and activities**

| Inputs | Activities | Outputs |
|--------|------------|---------|
| What goes in | What the program does | What it produces |

**Table 2: Expected outcomes**

| Short-term (0-3 months) | Mid-term (3-12 months) | Long-term (future goals) |
|-------------------------|------------------------|--------------------------|
| Immediate changes | Program goals achieved | Ultimate impact |

**Customize only three elements** for each program:

- **Activities**: The specific services this program provides
- **Mid-term outcomes**: The program's stated goals
- **Long-term outcomes**: The ultimate impact the program aims for

The other elements follow predictable patterns. This standardization makes evaluation manageable without sacrificing rigour.

**Long-term outcomes are aspirational targets.** They belong in the logic model to guide direction, but they are not directly measured. Focus on outputs through mid-term outcomes.

## Four Project Phases

### Phase 1: Planning and Initiation

**Focus:** Engage people and clarify scope

**Key tasks:**

- Engage the project sponsor, team members, and key decision-makers
- Define evaluation objectives and questions
- Clarify program activities (what the program actually does)
- Identify interest groups and their concerns

**Deliverables:**

- Clear roles and responsibilities assigned
- Agreed timelines and resources
- Defined evaluation questions
- Understanding of who cares about what

### Phase 2: Design and Development

**Focus:** Select measures and prepare tools

**Key tasks:**

- Define desired outcomes and potential risks
- Select measures for activities and participant goals
- Select measures for outcomes and interest group feedback
- Develop or adapt data collection tools

**Keep it simple.** The LogicalOutcomes framework includes standard tools that work for most programs. You don't need custom instruments.

**Deliverables:**

- Complete evaluation plan (adapted from Evaluation Planner output)
- Data collection tools ready to use
- Clear understanding of what data exists and what needs collection

### Phase 3: Implementation and Analysis

**Focus:** Collect data and share findings as you go

**Key tasks:**

- Collect data with attention to quality and coverage
- Design reports and visualizations early (don't wait until the end)
- Share emerging findings through interim reports
- Adjust collection if certain groups are underrepresented

**Report frequently.** Monthly feedback to staff enables continuous improvement. Don't wait for a final report.

### Phase 4: Communication and Discussion

**Focus:** Turn findings into action

**Key tasks:**

- Produce and discuss reports with staff and interest groups
- Focus discussions on recommended actions, not just findings
- Document changes made in response to findings
- Communicate results to broader audiences as appropriate

**Deliverables:**

- Final report with clear recommendations
- Evidence of discussions with staff about findings
- Documentation of actions taken or planned

## Data Collection Priorities

Prioritize data sources in this order:

### 1. Participant Tracker Data (Primary)

If the organization has a participant tracking system, use it as your primary data source:

- **Near 100% response rate** when the system is useful to staff
- **Collected as part of service delivery**, not a separate burden
- **Provides longitudinal data** about individual participants
- Include demographic information (with consent) to track equity

**Until an organization has solid participant tracking, they shouldn't invest in complex outcome measurement.**

### 2. Small-Sample Interviews (5-15 Participants)

Use when participant tracker data is insufficient:

- Targeted questions for specific diagnostic purposes
- Rich qualitative insight
- Most useful for understanding *why* patterns exist

### 3. Staff Meeting Notes

- Document how feedback is reviewed and discussed
- Track service changes made in response
- Evidence that the improvement cycle is working

### 4. Observation and Audit

- Verify program quality and fidelity
- Check that data collection is happening as planned

## The Participant-Informed Practice Cycle

Include this cycle in every evaluation plan:

```text
1. Collect feedback (during service delivery)
         ↓
2. Review in monthly staff meetings
         ↓
3. Discuss with decision-makers
         ↓
4. Change services in response
         ↓
5. Document changes → return to step 1
```

This is what turns data into service improvement. If nothing changes, the cycle isn't working.

## How Many Measures?

**Total: 8-11 indicators** distributed across three categories:

| Category | Default | Range | Includes |
|----------|---------|-------|----------|
| Output indicators | 4 | 3-5 | Program fidelity (did we deliver it right?) |
| Short-term outcome indicators | 4 | 3-5 | Participant experience (how did participants experience it?) |
| Mid-term outcome indicators | 3 | 2-4 | Goal achievement |
| **Total** | **11** | **8-11** | |

**Important**: Maintain at least 2 indicators in each category to preserve feedback loops.

**Action requirement**: Each monthly staff meeting should result in at least one documented service improvement or issue for investigation.

Of these, only **3-5 should be discussed in regular staff meetings**.

### Rules of Thumb

- **If staff can't explain why we track it, cut it.**
- **If we've never changed anything based on it, cut it.**
- **If it takes more than 30 seconds to record per participant, simplify it.**

### Review Frequency

| Frequency | What to Track | Purpose |
|-----------|---------------|---------|
| Weekly | Operational alerts (2-3) | Catch problems early |
| Monthly | Key indicators (3-5) | Staff discussion and action |
| Quarterly | Aggregated outputs | Management review |
| Annually | Mid-term outcomes | Strategic assessment |

## Working with Funders

Funders sometimes request approaches that conflict with best practice—typically mass surveys or overly complex frameworks.

**When this happens:**

1. **Acknowledge** the funder requirement in the evaluation plan
2. **Recommend** the LogicalOutcomes approach as the primary method
3. **Offer** the funder-required method as supplementary
4. **Document** the limitation clearly

**Example language:**

> "Note: Mass surveys typically have low response rates in community settings (10-30%) and may not yield statistically valid or actionable data. We recommend supplementing any survey with participant tracker analysis and small-sample interviews."

**Never silently compromise.** Make conflicts explicit so they can be discussed.

### Communicating Capacity Honestly

When funders ask "how do you know your program works?" the answer is:

- We have strong evidence that participants are achieving their stated goals
- Staff are making specific service improvements based on participant feedback
- Our data quality is high because staff actually use it
- We can show documented changes we've made in response to what we learned

Good data on what you can measure beats bad data on what you wish you could measure.

## Checklist Before Finalizing Any Plan

- [ ] Clear project ownership established (Sponsor, Owner, Manager roles)
- [ ] Logic model uses two 3-column tables (not one 6-column table)
- [ ] Evaluation framework tables use 4 columns: Outputs/Outcomes | Indicators | Data Source/Method | Frequency
- [ ] Primary data source is participant tracker (not surveys)
- [ ] Every measure connects to a specific action or decision
- [ ] Participant-informed practice cycle included
- [ ] Sustainable reporting schedule (monthly/quarterly/annual)
- [ ] Correct terminology (interest groups, participants, small-sample interviews)
- [ ] 8-11 total indicators with at least 2 in each category
- [ ] Conflicts with framework documented explicitly

## What We Do / Don't Do

### We DO:

- Use the standard logic model (two 3-column tables)
- Recommend participant tracker data as the primary source
- Suggest small-sample interviews (5-15 participants) when tracker data is insufficient
- Include the participant-informed practice cycle in every plan
- Tie every measure to a specific program change action
- Frame evaluation as an intervention that improves services

### We DON'T:

- Recommend mass surveys
- Recommend "client satisfaction surveys"
- Create custom logic model structures
- Suggest measures without specifying how they lead to improvement
- Recommend data collection that isn't integrated into service delivery
- Over-engineer with complex multi-domain frameworks
